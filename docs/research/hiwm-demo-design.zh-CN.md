# HIWM Demo 简版设计

> 版本：V2.1  
> 日期：2026-07-16  
> 用途：确定产品 Demo 做什么、页面怎么展示、首版实现哪些功能

## 1. 产品要做什么

这是一个装在机器人背后的**人际互动预测系统**。

机器人通过摄像头和麦克风观察正在与它交流的人，把四类信息放在一起：

1. 对方说了什么；
2. 对方说话时的语速、音高、音量和停顿变化；
3. 对方可见的面部运动、头部姿态和视线方向；
4. 机器人对这个人的初始了解和历史互动记录。

系统据此维护一个会不断更新的“用户互动画像”，推演机器人接下来采取不同回应可能带来的结果，并自动选择和执行当前最合适的行动。整个过程不等待操作员点击。真实回应出现后，系统再比较“预测和实际”，更新对这个人的理解。

第一版的“行动”只指机器人下一句采用什么沟通策略、具体说什么，或者暂时等待；暂不包含移动、抓取等实体动作。

一句话：

> **机器人在开口之前，先模拟不同回应可能把互动带向哪里；开口之后，再用真实反馈修正判断。**

完整闭环：

```text
看见与听见
→ 更新当前用户画像
→ 推演 3 个候选行动
→ 预测每个行动的后续结果
→ 锁定并执行一个行动
→ 观察用户真实回应
→ 对比预测与实际
→ 回写用户画像
```

## 2. Demo 展示形式

首版不需要在主画面放数字人。

主画面就是**机器人摄像头看到的真人画面**。它可以来自真实机器人，也可以先用电脑摄像头模拟。页面本质上是机器人的“实时思考看板”。

数字人只作为可选项：没有实体机器人时，用数字人代表机器人说话；有实体机器人后，直接把同一套系统接到机器人即可。


### 3.2 左侧：用户互动画像

分成三部分：

- **已知信息**：用户主动提供的背景、偏好、历史对话和已经确认的事实；
- **当前关键判断**：系统现在认为最可能影响本轮互动的因素；
- **仍然未知**：为了决定下一步，还需要确认哪些信息。

每条判断显示：

- 支持这个判断的主要证据；
- 不确定性；
- 相比上一轮新增或被推翻了什么。

首版不预设“顾虑程度、信息充分度、推进准备度”这类通用指标。不同业务场景需要先单独定义自己的状态字段；定义没有完成前，界面只展示自然语言判断、证据和不确定性。

画像不是给人贴永久标签。它只描述当前任务中的工作假设，并且允许被后续证据推翻。

### 3.3 中间：机器人视角

中间是整个页面的视觉中心，也是最需要做出技术展示感的区域。

#### 真人视频上的分析叠层

- 用细线标出人脸检测框；
- 底层可以读取完整面部 landmarks，界面只挑选约 60–80 个眉眼、鼻翼、嘴部和下颌关键点显示，避免遮住人脸；
- 只保留少量关键连线，不用完整网格遮住人脸；
- 在画面边缘显示头部左右转动、抬头低头和跟踪质量；
- 当某个可见动作明显变化时，在对应区域短暂出现标记，例如 `browInnerUp ↑`、`jawOpen ↑`；
- 点阵使用冷色、低透明度，发生变化时短暂发亮，随后恢复。

这些标记展示的是底层视觉特征，不直接显示“高兴、紧张、愤怒”等心理标签。

#### 语气变化的展示

视频下方放一条持续滚动的声音轨道：

- 第一层：实时波形和当前是否正在说话；
- 第二层：音高曲线，展示相对本人本轮基线的升降；
- 第三层：声音能量、语速和停顿片段；
- 明显变化生成时间戳证据，例如“语速较本轮前段上升”“回答前停顿 2.1 秒”。

不要把音高、音量或停顿直接换算成“情绪分数”。

#### 语言内容的展示

- 实时字幕逐字出现；
- 与任务相关的词句自动高亮，例如“实施太慢”“预算要审批”；
- 高亮内容自动生成事实、顾虑、承诺或未知项；
- 点击证据时可以定位回对应的视频和声音时间点。

最终在视频下方形成三条同步证据流：

| 证据 | 页面展示示例 |
|---|---|
| 视觉 | “头部转向屏幕”“browInnerUp 上升”“面部跟踪质量 91%” |
| 语气 | “语速较本人本轮基线上升 12%”“停顿 2.1 秒” |
| 内容 | “再次询问实施周期”“明确表示预算尚未确定” |

三条证据流在用户结束一轮发言后汇入左侧“当前关键判断”，形成一个短暂的汇聚动画。视觉和语气是辅助证据，语言内容与业务事实应当拥有更高权重。

### 3.4 右侧：行动预测树

根节点是当前用户画像，下面同时推演三种机器人行动。

每条分支只展示四项：

- 机器人准备采取的行动；
- 用户下一步可能回应；
- 达成本轮目标的预测概率；
- 风险和不确定性。

示例：

| 候选行动 | 预测结果 | 目标概率 | 风险 |
|---|---|---:|---|
| A：继续介绍功能 | 用户可能要求稍后看材料 | 42% | 低 |
| B：直接解释完整方案 | 可能回答顾虑，但信息增益有限 | 58% | 中 |
| C：提出一个小型试点 | 更可能确认约束并约下一步 | 81% | 低 |

系统在用户结束一轮发言后自动完成以下动画：

1. 当前关键判断成为根节点；
2. 三条候选行动同时向右展开；
3. 每条分支的预测结果和概率逐步出现；
4. 被选择的分支发亮并显示 `AUTO SELECTED`；
5. 其他分支变灰，但继续保留供观众对比；
6. 机器人立即说出所选回应，页面不出现确认按钮。

如果所有行动都低置信或高风险，系统自动选择“继续澄清”或“暂时等待”，而不是暂停等待人工点击。

概率必须标记为“模型预测”。在模型完成真实数据校准前，Demo 数字还要标记为“演示估计”，不能把它包装成已经验证的准确率。

### 3.5 底部：反馈闭环

底部时间线是 Demo 最重要的可信度设计：

```text
预测生成
→ 自动选择并锁定
→ 机器人立即回应
→ 用户真实回应
→ 预测命中 / 部分命中 / 偏差
→ 当前画像更新
→ 下一轮预测树刷新
```

这里的“锁定”是将行动前的预测写入记录，不是等待人类批准。这能让观众看到系统不是事后解释，而是在行动之前做出预测。

## 4. 三分钟演示流程

1. **开始会话**：页面加载用户初始信息，摄像头和麦克风开启。
2. **用户说话**：字幕、语音特征和面部运动同时更新。
3. **形成判断**：左侧画像根据本轮新证据形成当前关键判断和未知项。
4. **展开预测树**：右侧出现三种机器人行动及预测概率。
5. **自动执行**：系统高亮推荐分支、锁定预测，并立即生成机器人回应。
6. **机器人回答**：通过扬声器、数字人或实体机器人自动说出回应，全程无需点击。
7. **用户继续回应**：系统显示实际回应与预测的匹配和偏差。
8. **画像回写**：左侧关键判断和未知项发生变化，右侧生成下一轮预测树。

演示最好连续走两轮。第一轮证明“会预测”，第二轮证明“会根据真实反馈改变判断”。

## 5. 首版必须实现的功能

| 优先级 | 功能 | 首版要求 |
|---|---|---|
| P0 | 摄像头和麦克风采集 | 能读取电脑摄像头；后续可换成机器人摄像头 |
| P0 | 实时字幕 | 显示临时字幕和最终字幕 |
| P0 | 面部运动采集 | 关键点、头部姿态、可见动作和跟踪质量 |
| P0 | 语音特征采集 | 语速、音高、能量、停顿和相对个人基线变化 |
| P0 | 内容理解 | 提取需求、顾虑、承诺、问题和未知信息 |
| P0 | 用户画像 | 保存初始信息，维护当前关键判断、证据、未知项和不确定性 |
| P0 | 候选行动 | 每轮生成 3 个不同的结构化行动 |
| P0 | 结果预测 | 为每个行动输出可能回应、目标概率和风险 |
| P0 | 自动行动选择 | 根据目标、风险和信息增益选定行动，不需要操作员确认 |
| P0 | 自动执行与锁定 | 执行前记录预测、时间和当时信息，然后立即触发机器人说话 |
| P0 | 反馈更新 | 对比真实回应，刷新画像与预测树 |
| P0 | 会话回放 | 能回放证据、预测、实际回应和画像变化 |

## 6. 最小系统结构

```mermaid
flowchart LR
    A["机器人摄像头与麦克风"] --> B["实时音视频"]
    B --> C["视觉、语音、文本证据"]
    C --> D["用户画像与当前关键判断"]
    D --> E["三个候选行动"]
    E --> F["结果预测与概率"]
    F --> G["自动行动选择"]
    G --> H["LLM 生成具体说法"]
    H --> I["TTS / 数字人 / 实体机器人"]
    I --> J["用户真实回应"]
    J --> C
```

需要说明：开源项目可以完成音视频、感知、记忆、图表和机器人接口，但**没有一个现成项目可以直接完成整个 HIWM**。真正需要自己构建的产品核心是：

1. 不同业务场景下关键状态或工作假设的定义与更新方法；
2. 不同行动下的结果预测模型；
3. 行动评分与选择方法；
4. 预测与真实回应之间的学习闭环。

## 7. 可参考的开源项目

### 7.1 优先参考完整项目

首选参考 **OpenAvatarChat**。它已经把实时通信、VAD、ASR、LLM、TTS 和数字人 Avatar 串成可替换的完整流程，也包含实时音视频和数字人动画。第一版没有必要重新搭一遍语音合成和数字人链路。

需要做的主要改造是：

1. 保留它的实时对话和数字人能力；
2. 把前端改成本文定义的“机器人思考看板”；
3. 把普通 LLM 回答节点替换成 HIWM 的画像、推演、自动决策和反馈节点。

| 优先级 | 项目 | 主要用途 |
|---|---|---|
| 首选 | [OpenAvatarChat](https://github.com/HumanAIGC-Engineering/OpenAvatarChat) | 完整实时对话与数字人框架；Apache-2.0，各插件和模型另行核验 |
| 备选 | [LiveTalking](https://github.com/lipku/LiveTalking) | 如果数字人推流、口型或打断效果不满足需求，用它替换数字人链路 |
| 无数字人版 | [LiveKit Agent Starter React](https://github.com/livekit-examples/agent-starter-react) | 如果主画面只展示机器人摄像头和真人，用它做实时音视频与字幕骨架 |

OpenAvatarChat 是集成框架，不代表所有模型权重都打包在一个许可证里；正式商用仍需分别检查所选 ASR、TTS、Avatar 和人物素材。

### 7.2 完整项目能力不足时，再补这些组件

| 需要补的能力 | 推荐项目 | 用途与限制 |
|---|---|---|
| 实时面部点阵 | [MediaPipe](https://github.com/google-ai-edge/mediapipe) | 浏览器实时 landmarks、blendshape 和头部变换；Apache-2.0，适合作为首版视觉叠层 |
| 更深入的面部动作研究 | [OpenFace](https://github.com/TadasBaltrusaitis/OpenFace)、[LibreFace](https://github.com/ihp-lab/LibreFace) | Action Unit、视线与表情研究；公开版本有商业限制，只作为研究和离线对照 |
| 可商用的基础语气特征 | [Silero VAD](https://github.com/snakers4/silero-vad) + [librosa](https://github.com/librosa/librosa) | 说话起止、停顿、音高、能量和语速；MIT / ISC，需要自己做实时窗口和个人基线 |
| 标准化语气研究特征 | [openSMILE](https://github.com/audeering/opensmile) | eGeMAPS 等声学特征，适合研究 Benchmark；公开版本不能直接用于商业产品 |
| 中文字幕效果不足 | [FunASR](https://github.com/modelscope/FunASR) | 中文流式 ASR、时间戳和标点；框架 MIT，具体模型单独核验 |
| 多语言字幕效果不足 | [WhisperLive](https://github.com/collabora/WhisperLive) + [faster-whisper](https://github.com/SYSTRAN/faster-whisper) | 近实时多语言转写；MIT，模型权重许可另查 |
| 概率状态与分支预测 | [pgmpy](https://github.com/pgmpy/pgmpy) | 贝叶斯网络、动态贝叶斯网络和短期推演；MIT，状态和参数仍需自己定义与校准 |
| 预测树与概率动画 | [React Flow](https://github.com/xyflow/xyflow) + [Recharts](https://github.com/recharts/recharts) | 分支树、选中路径、概率条和历史曲线；均为 MIT |
| 跨会话用户画像 | [Memobase](https://github.com/memodb-io/memobase) 或 [Mem0](https://github.com/mem0ai/mem0) | 保存已确认事实和历史事件；Apache-2.0，不能代替 HIWM 当前判断 |
| 自动流程编排 | [LangGraph](https://github.com/langchain-ai/langgraph) | 串联观察、画像、推演、自动执行和反馈；MIT，不负责预测本身 |
| 实体机器人接口 | [rosbridge_suite](https://github.com/RobotWebTools/rosbridge_suite) | 用 WebSocket 连接 ROS 的摄像头、扬声器和动作；BSD-3-Clause |

没有必要一开始把以上项目全部接入。先跑 OpenAvatarChat；只有某个模块效果不够，再用对应项目替换或增强。

## 8. 推荐的首版技术组合

最快的路线是以 OpenAvatarChat 为底座，而不是从零拼装每个模块：

```text
底座：OpenAvatarChat
保留：RTC + VAD + ASR + LLM + TTS + Avatar

新增前端：
- 中间机器人视角视频
- 面部点阵、语气曲线、文本高亮
- 左侧用户画像与当前关键判断
- 右侧自动预测树与概率
- 底部预测—实际—更新闭环

新增后端：
- HIWM 用户画像与证据融合
- 三个候选行动的结果预测
- 自动行动选择与立即执行
- 真实回应回写与下一轮更新

补充组件：
- MediaPipe：面部点阵
- Silero VAD + librosa：语气变化
- React Flow + Recharts：预测树和概率动画
- pgmpy 或自研模型：概率推演
- rosbridge_suite：以后接实体机器人
```

首版开发顺序：

1. 跑通 OpenAvatarChat 的实时对话和数字人；
2. 重做前端看板，并加入真实摄像头、面部点阵和语气曲线；
3. 打通画像更新、三分支预测、自动行动和反馈回写；
4. 获得真实数据后，再训练和校准专门的预测模型；
5. 最后把数字人执行接口替换或扩展为实体机器人接口。

第一版可以用“规则 + LLM 结构化输出”跑通产品闭环，但预测概率必须标为演示估计。真正的模型版本，需要用真实的“状态—行动—回应—结果”数据训练和校准。

## 9. 最终效果

投资人或 B 端客户看完以后，只需要记住：

> **中间是真人和机器人的实时互动；左边是机器人正在更新的用户理解；右边是它对不同回应后果的推演；底部证明它会根据真实反馈修正自己。**
