# HIWM 项目架构与演进边界

本文描述 HIWM Interaction Engine `0.2.x` 的实际代码结构、数据流和后续演进原则。它是当前实现的工程基线，不是对未来基础模型能力的承诺。

## 项目定位

HIWM 是位于实时音视频系统、大模型和机器人执行器之间的“人际互动决策层”。它维护可被后续证据检验的互动状态，在行动前比较三条候选沟通路径，锁定所选预测，再用真实回应更新工作信念。

系统刻意不建立未经证实的“完整人格模型”，也不把人脸、音高或停顿直接解释成隐藏情绪。当前数字肖像是“具体对象 × 当前任务 × 当前情境 × 有限时间窗口”中的动态互动画像。

## 代码架构

| 层 | 主要位置 | 当前职责 |
| --- | --- | --- |
| 交互前端 | `frontend/src/renderer/` | 知情同意、设备生命周期、实时画面、派生特征、互动画像、三分支预测和回放 |
| 服务与 API | `src/service/` | FastAPI、WebRTC、运行时配置、会话账本、机器人适配 API |
| 实时管线 | `src/chat_engine/` | Handler 装配、流/信号管理、会话生命周期和取消传播 |
| 多模态感知 | `src/handlers/asr/`、前端 perception store、摄像头帧 | final ASR、可观察视频/语音派生值和证据时间边界 |
| 世界模型 | `src/handlers/hiwm/world_model.py` | 结构化内容信号、工作信念、三条行动条件化预测和上一轮反馈 |
| 规划器 | `src/handlers/hiwm/planner.py` | 依据目标、信息增益、风险和不确定性确定执行分支 |
| 预测账本 | `src/handlers/hiwm/ledger.py` | 在执行前以 JSONL、`fsync` 和 SHA-256 锁定预测 |
| 执行适配 | TTS Handler、`robot_action_api.py` | 本机语音执行，以及供外部机器人安全消费的只读命令流 |
| 第三方能力 | Git 子模块及既有 Handler | Avatar、VAD、TTS 等可替换组件；许可证分别遵从原项目 |

没有为了符合一份理想目录图而重写已工作的 Vue/FastAPI/Handler 架构。前后端、模型和执行边界已经通过契约分离；大规模改成 React/Next.js 会增加回归风险，却不会提升 HIWM 核心能力。

## 一轮交互的数据流

```text
明确同意
  → 摄像头 / 麦克风 / 用户确认背景
  → VAD + final ASR，冻结当前证据窗口
  → 内容信号 + 工作信念 + 三条候选行动
  → 硬契约与安全边界校验
  → 确定性规划器选择行动
  → JSONL + fsync + SHA-256 预测锁
  → TTS / 机器人只读行动接口
  → 下一轮真实回应
  → 匹配、部分匹配、偏差或无法判断
  → 更新可撤回的互动画像
```

前端派生时间线和服务端预测账本是两个数据面：前者用于本地可视化与导出，后者用于证明预测先于行动发生。默认都不保存原始音频、视频或完整人脸点位。

## 核心数据结构

- `ObservationSnapshot`：当前 final ASR、可选摄像头证据和已授权历史；每项都有时间、哈希和来源。
- `ContentSignal`：需求、顾虑、承诺、问题和未知项，只能引用当前真实证据。
- `BeliefItem`：已知、工作假设或未知；包含置信度、证据和本轮变化。
- `CandidateAction`：策略、准备说出的文本、预测回应、预测状态变化、目标/风险/不确定性/信息增益。
- `PlannerDecision`：固定公式、权重、每条分支分量和最终选择。
- `LockedPredictionRecord`：完整预测与哈希锁；任何字段被修改后校验都会失败。
- `RobotCommand`：从已验证账本派生的只读语音命令，使用 `session_id:turn_id` 去重。

## 从方案文档选择性吸收的内容

已实现或强化：

- 摄像头、麦克风和文本的实时输入；
- 动态 Human State / Digital Twin 的安全版本，即有证据和不确定性的互动画像；
- 三条行动条件化未来路径；
- 预测与真实回应的反馈闭环；
- 认知过程、证据、锁定和画像变化回放；
- 模型、感知和执行接口可替换；
- 面向实体机器人的稳定 `v1` 读取接口。

有意不直接采用：

- 不把系统改成普通聊天机器人；
- 不凭单帧画面或语气输出“真实情绪”“人格类型”“信任度”等伪精确标签；
- 不在缺少真实标注数据时训练或宣称已校准的专用模型；
- 不让通用 HTTP 接口直接控制机器人移动、抓取等高风险动作；
- 不为追求目录一致性重写已经通过验证的技术栈。

## 分阶段路线

### 已具备的 MVP

实时媒体、final ASR、结构化三分支、服务端规划、预测锁、TTS、反馈更新、会话回放、知情同意和多会话管理已有实现及自动化覆盖。

### 核心能力下一步

1. 用经同意的真实轨迹建立离线评估集，报告 NLL、Brier Score、ECE 和行动排序结果。
2. 为明确业务场景定义状态字段和结果标签，而不是使用一套伪通用心理量表。
3. 验证语音/视频相对纯文本的增量价值，并做缺失模态和跨设备消融。
4. 增加人工覆盖、机器人回执和失败恢复协议。

### 高级能力

只有在真实数据、设备安全和合规评估完成后，才考虑多步轨迹模型、受控在线更新、ROS/厂商 SDK 执行器和实体动作。机器人自身仍必须保留限速、碰撞保护和急停，HIWM 不取代设备控制安全层。
